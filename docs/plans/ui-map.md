# План A — Карта интерфейса из скринов Figma

> Долговременный дизайн-документ. Не временный план: описывает, как готовить и
> поддерживать корпус «где что находится в интерфейсе». Возвращаться сюда при
> доработке генератора и при изменениях интерфейса.

## Цель

Дать боту понимание расположения элементов интерфейса («где включить сверку»,
«что значит восклицательный знак», «куда нажать»). Retrieval текстовый, поэтому
скрины не индексируются как изображения — они распознаются в текст и кладутся в
тот же RAG-корпус, что и разделы из Google Docs.

Ключевое решение: **мультимодальный поиск не делаем.** Скрин — исходник для
текстового описания, а не объект поиска.

## Источник скринов — живой тестовый сервер, не Figma

Скрины снимаются с **реального работающего сервиса** (тестовый сервер с
заполненными данными), а не из Figma. Причина: в Figma дизайн местами устаревший
и неоднородный, её пришлось бы вручную чистить и переименовывать. Живой продукт
по определению актуален, а имена экранов генерятся автоматически из маршрутов.
Figma остаётся **запасным** источником только для экранов, которых ещё нет в
проде.

Аккаунт с тестовыми данными есть только на тестовом сервере — сбор скриншотов
выполняется именно там (Playwright логинится под демо-аккаунтом и обходит UI).

### Стек сервиса и откуда взята карта обхода

Сервис `github_sellerdata` — **Laravel + Inertia.js + Vue 3**. Карта экранов
выведена из кода (ветка `develop`):

- `routes/web.php` — URL → Inertia-страница;
- `resources/js/Pages/*.vue` — страницы, вкладки, модалки, состояния.

Результат анализа зафиксирован в `crawl_map.yaml` (см. ниже) — декларативный
список экранов/состояний с путями и селекторами для Playwright.

**Важные факты из кода (влияют на кравлер):**

- **`data-testid` в коде НЕТ** — селекторы строятся на `id`, классе и видимом
  тексте (`#loginEmail`, `#pills-products-tab`, `[data-target='#filterModal']`).
  Это самый хрупкий момент: часть кликов, возможно, придётся докрутить после
  первого прогона.
- Логин: `/login` → `#loginEmail` + `#loginPassword` + `.submit_button`, успех
  редиректит на `/dashboard`.
- Группировки (по артикулу / баркоду) — это **Select2** (`#group-by-sku`,
  `#groupOptions`), кравлеру нужен помощник «открыть дропдаун + кликнуть опцию»,
  а не простой клик.
- Режимы дашборда — через query-параметр `?tab=grid` / `?tab=chart`.
- Модалки Bootstrap открываются кликом по элементу с `data-target="#id"`.
- 6 пользовательских разделов: Дашборд, Товары, Расходы, Выкупы, Склад,
  Настройки (+ Тариф, Подключение). Покрыты в `crawl_map.yaml`.

## Расположение

Офлайн-генератор рядом с существующим `build_rag_corpus.py`, по той же
конвенции `artifacts/`. Runtime-сервисы (`bot`, `vector-base`, `chroma`) и
`docker-compose.yml` не меняются.

```
vector-base/artifacts/ui_map/
├── build_ui_map.py          # генератор (по образцу build_rag_corpus.py)
├── crawl_ui.py              # кравлер Playwright: читает crawl_map.yaml → screens/*.png
├── crawl_map.yaml           # ГОТОВО: карта обхода (выведена из кода github_sellerdata)
├── README.md                # локальная инструкция по запуску
├── manifest.json            # состояние: screen → doc_id, hash, modified_time, overrides
├── screens/                 # ВХОД: PNG-скрины (в .gitignore, в репо не едут)
├── sources/                 # сырые vision-описания (JSON) — ТОЧКА ВЫВЕРКИ (в git)
└── ui_map_chunks.jsonl      # ВЫХОД: чанки в общей схеме корпуса (в git)

`crawl_map.yaml` уже создан на основе анализа кода. Структура файла:
`base_url`, `login` (path + steps с селекторами), `default_wait`, и список
`screens` — каждый с `key` (= схема `раздел__подраздел__состояние`), `path`,
опциональными `actions` (клики/select2 для открытия вкладок/модалок/состояний),
`ui_state` и `parent_screen` для модалок. Адрес стенда и креды демо-аккаунта —
через переменные окружения (`SELLERDATA_BASE_URL`, `SELLERDATA_DEMO_EMAIL`,
`SELLERDATA_DEMO_PASSWORD`), в файле не хранятся.
```

`.gitignore`: добавить `vector-base/artifacts/ui_map/screens/`. В git едут только
генератор, выверенные `sources/*.json`, `manifest.json`, итоговый JSONL.

## Конвенция имён экранов

Имя файла → `doc_id` и `section_path` (выводятся автоматически, вручную имена не
придумывать — только держать схему). Разделитель сегментов — `__`, глубина
произвольная:

```
<раздел>__<подраздел>[__<состояние>].png
```

Примеры:

```
дашборд__плитки.png                         → doc_id "ui:дашборд:плитки",  section_path ["Дашборд","Плитки"]
товары__карточка.png                        → базовый экран
товары__карточка__модалка-редактирование.png → doc_id "ui:товары:карточка:модалка-редактирование",
                                               section_path ["Товары","Карточка","Редактирование"]
настройки__интеграции.png                   → doc_id "ui:настройки:интеграции"
```

Экраны, ломающие схему, переопределяются вручную в `manifest.json`
(имя файла → doc_id + section_path). Это исключение, не правило.

### Платформа

Десктоп — базовый слой, `metadata.platform: "desktop"`. Мобайл подключается
позже и только для экранов с реально другой вёрсткой — отдельным чанком
`platform: "mobile"` под тем же экраном.

### Маркетплейс (WB / Ozon / Yandex Market)

Маркетплейс — **измерение, как платформа**, а не отдельные экраны. В коде это
активный магазин (`store.account.data_source`: `wildberries`/`ozon`/
`yandex_market`); один и тот же URL рендерится по-разному. Переключение — через
выпадашку магазинов в шапке (`changeAccount`, селектор `.user_account_status`).

Правило — **базовый слой + только расхождения** (как у мобайла):

1. Кравлер снимает каждый экран под каждым подключённым магазином →
   `товары__список@wb.png`, `товары__список@ozon.png`.
2. **Автодедуп по перцептивному хэшу:** идентичные WB/Ozon → один чанк
   `metadata.marketplace: common`; различающиеся → отдельные чанки `wb`/`ozon`.
   Пограничные («почти одинаковые») → флаг на выверку.
3. Имя: суффикс `@<mp>` только когда экраны различаются (`товары__список@wb`,
   `товары__список@ozon`); одинаковые остаются `товары__список` +
   `marketplace: common`. Суффикс `@` (не `__`) — чтобы не путать с сегментом
   раздела при разборе имени.

Связь с ботом: planner уже различает маркетплейсы (переспрашивает «WB или
Ozon?» при расхождениях). Чанки с `metadata.marketplace` позволяют retrieval
предпочесть нужный маркетплейс, `common` работает всегда — отдельной ветки в
боте не требуется.

Конфигурация в `crawl_map.yaml`: блок `marketplaces` (id + подпись магазина в
выпадашке) и `switch_store` (как переключить). Требование: на демо-аккаунте
подключены оба магазина. Yandex Market уже есть в коде — подключается третьим
позже.

### Модалки / попапы / оверлеи

Базовый экран и всплывающее поверх него — это **разные `doc_id` с общим
префиксом** (префикс склеивает их в поиске, отдельные `doc_id` дают точечные
обновления). Различие фиксируется метаданными, вручную их вписывать не нужно —
vision проставляет сам по картинке:

- `metadata.ui_state`: `screen` | `overlay` | `modal`.
- `metadata.parent_screen`: ключ базового экрана для оверлеев/модалок
  (например `товары__карточка`).

Практика съёмки: попап снимать **со фоном раздела**, не вырезанным окном — так
сохраняется привязка «где это в интерфейсе» и расположение точнее.

## Контракт vision-распознавания

Один экран → один vision-вызов через `vector-base/app/openai_client.py` (тот же
прокси), модель из новой переменной `OPENAI_VISION_MODEL`. Промпт требует только
структурированный JSON. Схема `sources/<screen>.json`:

```json
{
  "screen_key": "дашборд__плитки",
  "screen_title": "Дашборд → Плитки",
  "platform": "desktop",
  "ui_state": "screen",
  "parent_screen": null,
  "elements": [
    {
      "name": "Кнопка «Больше»",
      "location": "внутри каждой плашки, в правом нижнем углу",
      "purpose": "раскрывает детализацию плашки",
      "appearance": "серая текстовая ссылка со стрелкой",
      "aliases": ["развернуть плашку", "подробнее", "детализация"]
    }
  ],
  "screen_summary": "Вкладка показывает плашки с продажами/прибылью за период..."
}
```

Промпт инструктирует: расположение писать словами пользователя (углы, столбцы,
верх/низ); иконки/цвета называть как их называет пользователь («восклицательный
знак», «красная плашка»); не выдумывать функциональность, которой не видно.

## Этап выверки

После распознавания генератор пишет `sources/<screen>.json` и останавливается.
Человек правит текст руками. Чанки собираются отдельным шагом только из
выверенных `sources/`. Распознавание никогда не идёт в базу напрямую.

## Схема чанка (контракт `load_rag_corpus.py`)

Каждый `element` + общий `screen_summary` → чанки. Обязательны `id`,
`embedding_text` (по нему ищут), `answer` (отдаётся ассистенту).

```json
{
  "id": "ui:дашборд:плитки:0003:<hash>",
  "doc_id": "ui:дашборд:плитки",
  "doc_title": "Дашборд → Плитки",
  "doc_modified_time": "<modified_time из manifest>",
  "chunk_index": 3,
  "section_path": ["Дашборд", "Плитки"],
  "block_type": "ui_location",
  "title": "Где находится кнопка «Больше»",
  "question_intents": ["где кнопка больше", "как раскрыть плашку", "куда нажать чтобы увидеть детализацию"],
  "keywords": ["больше", "развернуть", "детализация", "плашка", "дашборд"],
  "answer": "Кнопка «Больше» находится в правом нижнем углу каждой плашки на вкладке «Плитки» раздела «Дашборд». Она раскрывает детализацию плашки.",
  "source_facts": ["Кнопка «Больше» — в правом нижнем углу плашки", "раскрывает детализацию"],
  "embedding_text": "Раздел: Дашборд > Плитки. Кнопка Больше развернуть детализация плашки где находится правый нижний угол ...",
  "metadata": {"platform": "desktop", "ui_state": "screen", "parent_screen": null, "source": "ui_map", "screen_key": "дашборд__плитки"}
}
```

- `block_type: "ui_location"` — новый тип в существующей линейке.
- `embedding_text` собирается из name + location + aliases + section_path.

## manifest.json

```json
{
  "format": "sellerdata_ui_map_v1",
  "screens": {
    "дашборд__плитки": {
      "doc_id": "ui:дашборд:плитки",
      "section_path": ["Дашборд", "Плитки"],
      "png_sha256": "...",
      "modified_time": "2026-06-11T...",
      "platform": "desktop"
    }
  }
}
```

По `png_sha256` неизменённые экраны пропускаются (vision не дёргается). Здесь же
ручные `overrides`.

## CLI

```bash
# 0. собрать скрины с тестового сервера по crawl_map.yaml (Playwright)
python -m artifacts.ui_map.crawl_ui [--only дашборд__график]

# 1. распознать новые/изменённые скрины → sources/*.json (vision)
python -m artifacts.ui_map.build_ui_map recognize [--force] [--only дашборд__график]

# 2. (человек правит sources/*.json руками)

# 3. собрать ui_map_chunks.jsonl из выверенных sources/
python -m artifacts.ui_map.build_ui_map build
```

`recognize` и `build` разделены намеренно — между ними выверка. Шаг 0
(кравлер) автоматизирует сбор; при отсутствии скрина можно положить PNG в
`screens/` вручную (fallback из Figma).

## Правки в runtime vector-base (минимальные)

1. `load_rag_corpus.py` — принимать несколько JSONL (папку/список) и сливать в
   одну коллекцию при полной пересборке (`replace_corpus`).
2. Инкрементальная загрузка по `doc_id` — CLI `load-one` поверх
   `delete_document(doc_id)` + `add`. Для обновления одного экрана без
   переэмбеддинга всего корпуса.
3. Makefile: `make build-ui-map`, `make load-ui DOC_ID=...`; общая `make load-rag`
   сливает `rag_corpus` + `ui_map`.
4. `.env` / `.env.example`: новая `OPENAI_VISION_MODEL` (с комментарием).

## Сценарий «интерфейс перерисовали» (безболезненно)

1. Кладёшь обновлённый `screens/дашборд__плитки.png` (то же имя).
2. `recognize` — по hash перераспознаёт только этот экран.
3. Правишь `sources/дашборд__плитки.json`.
4. `build` — перегенерирует чанки этого `doc_id`.
5. `make load-ui DOC_ID=ui:дашборд:плитки` — точечно `delete`+`add`. Остальная
   база нетронута.

## Тесты (`vector-base/tests`)

- парсинг имени файла → `doc_id` + `section_path` (включая override и `__состояние`);
- сборка чанка из `source`-JSON соответствует контракту `_read_jsonl`
  (есть `id`/`embedding_text`/`answer`);
- skip по неизменному `png_sha256`;
- `load_rag_corpus` сливает несколько JSONL без коллизий `id`;
- инкрементальная загрузка одного `doc_id` не трогает остальные (мок Chroma);
- vision-вызов мокается, реальный OpenAI в тестах не дёргается.

## Документация

- `docs/architecture.md`: в раздел `vector-base` — про `artifacts/ui_map/`,
  `block_type: "ui_location"`, инкрементальную загрузку; в «Данные» — `screens/`
  вне git; новая переменная `OPENAI_VISION_MODEL` в «Конфигурации».
- `README.md`: короткий раздел «как обновить карту интерфейса».

## Порядок работ

1. **[готово]** `crawl_map.yaml` — карта обхода из кода `github_sellerdata`.
2. Кравлер `crawl_ui.py` на Playwright: логин + обход по `crawl_map.yaml` →
   `screens/*.png`. Помощник `select2` для группировок. Прогон на тестовом
   сервере, докрутка хрупких селекторов (data-testid в коде нет).
3. Скелет `build_ui_map.py` + парсер имён + manifest (без vision) + тесты на парсинг.
4. Vision-шаг `recognize` (мок в тестах) + контракт `source`-JSON.
5. Шаг `build` → JSONL + тесты на контракт схемы.
6. Правки `load_rag_corpus.py` (мульти-JSONL) + инкрементальная загрузка + тесты.
7. Makefile-цели.
8. Документация, `.gitignore`.
9. `make test`, `make health`, прогон на реальных скринах, выверка, загрузка.

## Статус первого прогона кравлера (2026-06-11)

Прогон на проде под демо-аккаунтом (`demo@sellerdata.ru`), оба магазина:
**снято 42 PNG = 21 экран × WB и Ozon.** Полный конвейер (логин → переключение
магазина → навигация → открытие модалки → скриншот) работает на живом сайте.
Различия WB/Ozon подтверждены визуально (разный состав сайдбара и контент) —
дедуп по маркетплейсам обоснован.

Технические правки, найденные на живом DOM (внесены в `crawl_ui.py`):
- `networkidle` → `domcontentloaded` (постоянный websocket pusher);
- логин-ожидание по элементу `.user_account_status`, а не по URL (Inertia SPA);
- `_safe_goto` с ретраем на `ERR_ABORTED` (гонка SPA-навигаций);
- переключение магазина: точечный селектор `a.user_auth_nav_item_link`;
- запуск кравлера — с отключённой песочницей (headless грузит JS-бандлы).

### Докрутка селекторов (выполнена) — повторный прогон чистый

После инспекции живого DOM карта приведена в соответствие: **повторный обход —
45 PNG, ноль пропусков.** Решения по 9 проблемным ключам:
- **Мобильные модалки фильтра** (`дашборд/товары/склад __модалка-фильтр`) — на
  десктопе скрыты (фильтр инлайновый). Закомментированы в карте, отнесены к
  будущему мобильному проходу.
- **Группировки** — реальные опции: у продаж «Бренду», у заказов «Заказам/
  Артикулу» (не «артикул/barcode»), селекты без id. Под-экраны убраны как
  переусложнённые: контрол виден на родительских экранах.
- **Чекбоксы товаров** — `без-себестоимости` починен (клик по `label[for=...]`);
  `скрытые` убран (на демо нет скрытых товаров).
- **Импорт** — на дашборде это не модалка, а страница `/dashboard/upd-import`
  (добавлена как `дашборд__импорт-упд`); «Импорт расходов» на десктопе скрыт,
  ключ убран.
- **`подключение__реклама-ozon`** — ограничен `marketplaces: [ozon]` (на WB 404).

В кравлер добавлены: per-screen фильтр `marketplaces:` и увеличенная пауза перед
снимком (полная отрисовка сайдбара/отложенных элементов).

### Полностраничные снимки + ожидание всех лоадеров (выполнено)

По требованию: (1) снимок всей прокручиваемой страницы, (2) без индикаторов
загрузки в кадре.
- `full_page: true` в конфиге (default), модалкам — `full_page: false` (оверлеи
  в viewport). CLI `--viewport-only` для отладки. `_wait` повторяется и после
  действий.
- `default_wait.hidden` — список контентных лоадеров (`.spinner-border`,
  `.loading-wrapper`, `.ring_loader`, `.loading-placeholder`,
  `.multi-select__loading`, `.btn-loader`); `.data_loader_block` исключён
  (постоянный виджет фоновой синхронизации). Отсутствующий селектор условие
  удовлетворяет мгновенно, поэтому список безопасен.
- Контрольный прогон: 45 PNG, ноль пропусков, ноль незакрытых лоадеров;
  full-page подтверждён (товары — 2102px вся таблица, модалка — 900px viewport).

### Финальный набор (после аудита и интерактивных состояний)

**51 PNG** (WB 25 + Ozon 26; у Ozon +1 — Ozon-only страница рекламы). Включает
исправленные `дашборд__диаграмма`, `подключение__магазин` и 3 интерактивных
состояния (плитка/строка-детализация, заказы по артикулу). Переключение
маркетплейса защищено: перед сменой магазина — возврат на чистый дашборд (сброс
открытых модалок), при сбое маркетплейс пропускается, а не падает весь обход.
Известный единичный случай: на одном Ozon-экране `.loading-wrapper` может не
исчезнуть за 15с (тяжёлая таблица) — проверить при выверке.

## Аудит экранов и интерактивные состояния (2026-06-11)

Сверка кадров с кодом выявила 2 неверных экрана (исправлены):
- `дашборд__диаграмма` — `?tab=chart` применяется только для business-тарифа;
  заменено на клик по вкладке «Диаграмма». Теперь показывает реальный график.
- `подключение__магазин` — `/connecting` редиректит на дашборд при валидном
  ключе; заменено на `/connecting?new`. Теперь форма подключения WB/Ozon.

Добавлены интерактивные состояния (раскрытия, не видимые на базовых экранах):
- `дашборд__плитка-детализация` — модалка разбивки плитки периода («Больше»).
- `дашборд__строка-детализация` — модалка детализации товара (стрелка
  `button.product-more-btn` в колонке «Инфо»).
- `дашборд__диаграмма__заказы__группировка-артикул` — заказы по артикулу.

Решения по остальным состояниям:
- **Тултипы** (`v-tippy`, 18 файлов) — скриншотами снимать ненадёжно (headless не
  удерживает hover). Их тексты извлекаются ИЗ КОДА (`v-tippy="'...'"`) и кладутся
  в корпус как факты «иконка X показывает подсказку: …». Это полнее и стабильнее.
  → отдельный шаг генератора (парсинг v-tippy из .vue).
- **Сортировки колонок** (~20 шт, `sortByColumn`) — меняют только порядок строк,
  новой информации нет. Отдельно не снимаем; факт «таблицу можно сортировать по
  колонкам X, Y, Z» добавляется текстом (список колонок — из кода).

## Распознавание и сборка выполнены (2026-06-15)

Распознавание сделано НЕ через OpenAI, а агентами Claude (Sonnet) — по просьбе
пользователя и из-за того, что прокси OpenAI доступен только внутри Docker.
8 параллельных агентов прочитали 99 скринов и записали `sources/*.json` по схеме;
качество выверено на выборке (точные подписи, иерархия рекламы, состояния).
Исправлена коллизия имён NFC/NFD у двух рекламных файлов.

`build_ui_map build` собрал **`ui_map_chunks.jsonl` — 787 чанков** + домержены
`custom_facts.jsonl`. Все чанки валидны (id/embedding_text/answer), без дублей id.

Дедуп WB/Ozon: автослияние по тексту не сработало (агенты пишут независимо), но
визуальный анализ (dHash) показал, что ~половина пар почти идентичны. Решение —
курируемый набор `COMMON_KEYS` в build_ui_map: маркетплейс-агностичные модалки/
формы (тариф, подключение магазина, модалки расходов/настроек, период, детализации)
и рекламные кадры (Ozon — копии WB, данных Ozon-рекламы нет) схлопнуты в один
набор `common`. Списки/таблицы (товары, настройки-магазины, расходы-список и т.п.)
оставлены по МП — там данные различаются. Итог: 985 → **787 чанков**
(20 common-экранов, 59 МП-специфичных групп). Без риска ложного слияния.

`load_rag_corpus.py` доработан: принимает НЕСКОЛЬКО JSONL (разделы + карта), с
проверкой коллизий id. Makefile `load-rag` грузит оба (rag_corpus 90 + ui_map 985
= 1075 чанков, пересечений id нет). `build` теперь работает на чистом stdlib
(импорты OpenAI вынесены в recognize).

ОСТАЛОСЬ (среда пользователя): реальная загрузка в ChromaDB — `make load-rag` при
поднятом стеке (`make up`) и активном SOCKS-прокси к OpenAI. С хоста выполнить
нельзя (стек не запущен, прокси — Docker-internal). Проверка после загрузки:
`make health` + тестовый `/search`.
